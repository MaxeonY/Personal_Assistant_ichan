## 1. 实现框架摘要

- 问题类型：播放器职责混杂、变更耦合高 -> 最终结论：播放器固定为四层（类型层 / 帧序定义层 / 资源加载与 pin 层 / token 化播放执行层），React 仅作为消费层。
- 问题类型：异步打断时播放链污染 -> 最终结论：所有打断统一走 `interrupt(oldToken) -> play(newParams)`，不允许旁路切换。
- 问题类型：多处维护状态导致链路不一致 -> 最终结论：`PetFullState` 作为唯一真实状态源，所有表现由同一状态快照派生。
- 问题类型：规则入口分散导致行为不可追溯 -> 最终结论：`dispatch(event)` 作为唯一规则入口，状态变更不允许直接写入。
- 问题类型：三层正交状态难以直接驱动播放 -> 最终结论：在规则层投影为单条当前播放路径，保留 movement/overlay 与 major 的职责边界。
- 问题类型：targeted_move 到达事件可能迟到误触发 -> 最终结论：`movement.arrive` 必须执行 `requestId` 防陈旧校验。
- 问题类型：hungry 语义与表现混淆 -> 最终结论：逻辑层保持 `flag/overlay`，渲染层使用独立 hungry 素材叠加，不并入 `MajorState`。
- 问题类型：drowsy 切出观感与契约口径不统一 -> 最终结论：统一为三段式，并固定短退出口径“4 源帧 + 目标态首帧自然衔接”。

## 2. 验证方法摘要

- 问题类型：不同层问题混在一起排查效率低 -> 最终结论：表现层问题、状态机问题、外部服务问题必须分层定位。
- 问题类型：过早接真实依赖导致定位噪声大 -> 最终结论：先验证事件触发链，再验证真实依赖。
- 问题类型：验证阶段切换无门槛导致回归失控 -> 最终结论：按“动画验证 -> 状态机验证 -> mock 集成验证 -> 真实集成验证”逐层推进。
- 问题类型：重复造验证脚本、结果不可横向比较 -> 最终结论：复用已验证有效的通用模式与模板，统一命名、统一链路表达、统一断言口径。
- 问题类型：验证壳异常被误判为内核缺陷 -> 最终结论：demo/unit/harness 壳问题与核心状态机问题分离判断、分离结论。
- 问题类型：打断行为描述不一致 -> 最终结论：按软/硬打断分类进行链路验证，避免混用口径。

## 3. 工程经验摘要

- 问题类型：状态切换首帧闪烁（冷解码） -> 最终结论：`SpriteSheet + background-position` 是唯一正确方向，参数调速不能替代渲染路径修正。
- 问题类型：显示盒在不同素材下失真/裁切 -> 最终结论：显示层按 metadata 自描述，不硬编码帧尺寸。
- 问题类型：动画推进偶发顿挫与冻结 -> 最终结论：统一单调时钟基准，避免 `Date.now()` 与 `performance.now()` 混用。
- 问题类型：看起来在走但没有真正位移 -> 最终结论：`roaming/targeted_move` 必须同时具备步态动画与窗口真实位移执行链。
- 问题类型：位移执行链正常但位移不生效 -> 最终结论：优先检查 capability 权限（如窗口位置写权限），不要先改状态机语义。
- 问题类型：点击经常被拖拽吞掉 -> 最终结论：归类为输入层手势冲突，使用阈值与抑制策略在输入层修复。
- 问题类型：非功能修复干扰业务迭代判断 -> 最终结论：性能/渲染/输入等非功能问题与业务逻辑问题解耦管理、解耦验收。
